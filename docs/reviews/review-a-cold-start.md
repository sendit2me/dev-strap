## Understanding Summary

DevStrap is a code generator that produces a complete Docker Compose development environment from a convention-driven directory structure. Its central mechanism is HTTPS mock interception: when the app container makes an HTTPS request to, say, `api.stripe.com`, Docker's internal DNS resolves that domain to a Caddy reverse proxy (via network aliases set in the generated `docker-compose.yml`). Caddy terminates TLS using a self-signed certificate (generated by a one-shot `cert-gen` container that reads all domains from `mocks/*/domains`), then proxies the decrypted request to a WireMock instance that returns canned responses from JSON stub files. The app trusts the custom CA via language-specific environment variables (`NODE_EXTRA_CA_CERTS`, `SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`). This means the app source code is byte-identical between dev and production -- no `isDev` branching.

The system has three source-of-truth inputs: `project.env` (stack config -- app type, database, ports, extras), `mocks/*/domains` + `mocks/*/mappings/*.json` (what to intercept and how to respond), and `templates/` (composable service definitions). Two generator scripts in `core/` consume these inputs: `core/caddy/generate-caddyfile.sh` produces `.generated/Caddyfile` and `core/compose/generate.sh` produces `.generated/docker-compose.yml` by concatenating template fragments with sed-based variable substitution. Everything under `.generated/` is ephemeral -- deleted on `stop`, rebuilt on `start`. `devstack.sh` is the monolithic CLI orchestrating generation, Docker lifecycle, mock management (new-mock, record, apply-recording, reload-mocks, verify-mocks), testing (Playwright in container), and a PowerHouse contract interface (`--options`, `--bootstrap`).

The developer workflow is: edit `project.env` and mock definitions, run `./devstack.sh start` (generates config, builds containers, starts stack), iterate on app code with live reload (file watching via `node --watch`, Air for Go, uvicorn `--reload`, cargo-watch for Rust), edit mock mappings and hot-reload them via `./devstack.sh reload-mocks`, run tests with `./devstack.sh test`, and tear everything down with `./devstack.sh stop`. The system also supports recording real API responses (`record`) and promoting them to mock mappings (`apply-recording`). A contract interface lets an external tool (PowerHouse) discover the catalog (`--options`) and generate a project non-interactively (`--bootstrap --config payload.json`).

## What Worked (from code analysis)

1. **Core architecture is sound.** The DNS alias + Caddy TLS termination + WireMock chain is a legitimate, well-designed approach to transparent mock interception. The certificate generation covers all mock domains as SANs, and each app template correctly sets the language-appropriate CA trust environment variable.

2. **Generation pipeline is clean.** The two generator scripts (`core/caddy/generate-caddyfile.sh` and `core/compose/generate.sh`) produce well-structured output. The Caddyfile generator correctly handles three app-type branches (PHP-FPM, frontend+backend, plain reverse proxy) and aggregates all mock domains into a single WireMock proxy block.

3. **Mock system is well-organized.** The directory convention (`mocks/<name>/domains` + `mocks/<name>/mappings/`) is intuitive and filesystem-visible. The example mocks demonstrate three WireMock patterns (simple, stateful via scenarios, conditional via bodyPatterns + priority) that are genuinely useful.

4. **Test infrastructure is containerized.** Running Playwright inside a container with the same CA trust means tests exercise the actual HTTPS interception chain, not a bypassed version. The test dashboard (busybox httpd serving results) is a nice low-overhead addition.

5. **Mock hot-reload works correctly.** `reload-mocks` calls WireMock's `/__admin/mappings/reset` endpoint, which re-reads from the mounted volumes. This avoids a full restart for mapping-only changes.

6. **Record/apply flow is thorough.** The `apply-recording` command correctly rewrites `bodyFileName` paths to account for WireMock's subdirectory-based mounting and uses Docker to fix file ownership from root to the host user.

7. **Contract validation is comprehensive.** The `validate_bootstrap_payload` function implements 11 validation checks in a single jq expression (contract field, version, project name format, unknown categories/items, required categories, single-selection enforcement, dependency checking with wildcard support, conflict detection, override key validation, and port collision detection). This is well-engineered.

8. **Template catalog is broad.** Five app types (node-express, php-laravel, go, python-fastapi, rust), two databases, four services (redis, mailpit, nats, minio), six tooling options, and three observability options. Each has a proper healthcheck defined.

## What Broke (bugs, issues, confusing behavior)

### Severity: High

**1. `EXTRAS_DEPENDS` is dead code -- extras never become app dependencies.**
File: `/home/i9user/Projects/DS/dev-strap/core/compose/generate.sh`, lines 94, 121-123.
The variable `EXTRAS_DEPENDS` is built up in the extras loop (adding each extra as a `depends_on` entry with `condition: service_healthy`) but is never referenced again in the file. It is never injected into the app service's `depends_on` block or any other service. This means extra services like Redis do not block the app from starting, which could cause connection failures if the app tries to reach Redis before it is healthy. This was likely intended to be appended to the app service definition but was lost during refactoring.

**2. PHP-Laravel `DB_CONNECTION` is hardcoded to `mysql`, breaks with PostgreSQL.**
File: `/home/i9user/Projects/DS/dev-strap/templates/apps/php-laravel/service.yml`, line 13.
`DB_CONNECTION=mysql` is always set, regardless of `DB_TYPE`. If a user selects `php-laravel` with `DB_TYPE=postgres`, the app will still try to connect using the MySQL driver. The compose generator does correctly substitute `${DB_PORT}` (5432 for postgres), but the connection driver will be wrong. The research document (`docs/research/05-stack-combinations.md`, line 1066) explicitly notes that PostgreSQL should use `DB_CONNECTION=pgsql`, but this was never implemented.

**3. App service has no healthcheck in any template.**
Files: All `templates/apps/*/service.yml` files.
None of the five app templates define a `healthcheck`. The web container's `depends_on` uses `app: condition: service_started`, not `service_healthy`. This means Caddy may start proxying before the app is ready, causing 502 errors. The `web` service itself has a healthcheck (`wget --spider http://127.0.0.1/`), but that health check will only pass once Caddy can reach the app -- creating a cascading dependency that relies on timing rather than explicit health signaling.

**4. `init --preset` is documented but does not exist.**
File: `/home/i9user/Projects/DS/dev-strap/devstack.sh`, function `cmd_init()` (lines 779-918).
The `cmd_init` function does basic interactive prompting but has no `--preset` flag handling. The QUICKSTART doc (lines 48-51) shows `./devstack.sh init --preset full-stack` as a valid command. The README (line 121) also references it. Running this would cause the shell to interpret `--preset` as the response to the "Project name" prompt (since `read -r` would capture it). The presets exist in `contract/manifest.json` but are only accessible via `--bootstrap`, not via `init`.

**5. The example app's HTML references nginx instead of Caddy.**
File: `/home/i9user/Projects/DS/dev-strap/app/src/index.js`, line 73.
The home page says: "DNS + nginx + WireMock intercept them and return mock responses." The system uses Caddy, not nginx. This is a stale reference from before the Caddy migration (documented in `docs/TODO.md` line 8 as completed).

### Severity: Medium

**6. `FRONTEND_TYPE`, `FRONTEND_SOURCE`, `FRONTEND_PORT` are not defined in the shipped `project.env`.**
File: `/home/i9user/Projects/DS/dev-strap/project.env`.
The compose generator references `${FRONTEND_TYPE:-}`, `${FRONTEND_SOURCE}`, and `${FRONTEND_PORT:-5173}` but these variables are not present in the default `project.env`. The generator uses `:-` defaulting for `FRONTEND_TYPE` and `FRONTEND_PORT` so it won't crash, but `FRONTEND_SOURCE` has no default protection. If a user added `FRONTEND_TYPE=vite` to `project.env` without also adding `FRONTEND_SOURCE`, the compose generator (line 87) would try to expand `${DEVSTACK_DIR}/${FRONTEND_SOURCE#./}` with an empty `FRONTEND_SOURCE`, producing a path pointing to the project root -- which would be mounted as the entire repo into the frontend container.

**7. `DB_ROOT_PASSWORD` is substituted into Postgres template but Postgres doesn't use it.**
File: `/home/i9user/Projects/DS/dev-strap/core/compose/generate.sh`, line 148.
The compose generator runs `sed "s|\${DB_ROOT_PASSWORD}|${DB_ROOT_PASSWORD}|g"` on the database template, but `templates/databases/postgres/service.yml` does not contain `${DB_ROOT_PASSWORD}` anywhere. PostgreSQL does not have a separate root password concept. This is harmless (sed finds nothing to replace) but suggests the generator was written for MariaDB and PostgreSQL was bolted on without clean-up.

**8. WireMock mapping path collisions across mocks are silently merged.**
File: `/home/i9user/Projects/DS/dev-strap/core/compose/generate.sh`, lines 48-50.
Each mock's mappings directory is mounted into WireMock at `/home/wiremock/mappings/<mock_name>/`. If two mocks define mappings with the same URL path (e.g., both `example-api` and `example-payment` define `POST /v1/items`), WireMock will load both and match whichever has higher priority (or return the first match at the same priority). There is no collision detection, no warning, and the `verify-mocks` command only checks domain reachability, not mapping conflicts. The docs do not mention this risk.

**9. `record` command requires manual network name knowledge.**
File: `/home/i9user/Projects/DS/dev-strap/devstack.sh`, line 583.
The recorder container joins the network `${PROJECT_NAME}_${PROJECT_NAME}-internal`. Docker Compose v2 names networks as `<project>_<network-name>`. But the `-p` flag sets the project name, so the network is `${PROJECT_NAME}_${PROJECT_NAME}-internal`. However, the actual app container needs to know the recorder's hostname (`${PROJECT_NAME}-recorder`) to route requests through it. The instructions in the "no mappings captured" fallback (line 617) tell the user to `wget` from `http://${PROJECT_NAME}-recorder:8080`, but the app's DNS resolution will still route mocked domains to Caddy/WireMock, not to the recorder. The recorder sits outside the normal interception chain -- to actually use it, the user would need to temporarily change how the app resolves the target domain, which is not documented.

**10. `handle_path` strips the path prefix in frontend+backend mode.**
File: `/home/i9user/Projects/DS/dev-strap/core/caddy/generate-caddyfile.sh`, line 96.
Caddy's `handle_path` directive strips the matched prefix before proxying. So a request to `localhost:8080/api/users` gets proxied to `app:3000/users` (the `/api` prefix is removed). This is a deliberate Caddy behavior, but it means the backend must NOT define its routes under `/api/*` -- it must define them at the root (`/users`, not `/api/users`). If a developer sets up Express routes as `app.get('/api/users', ...)`, they won't match because Caddy stripped the `/api` prefix. This is a significant developer footgun that the docs do not clearly warn about.

### Severity: Low

**11. The `init` command's HTTPS_PORT calculation is fragile.**
File: `/home/i9user/Projects/DS/dev-strap/devstack.sh`, line 835.
`HTTPS_PORT=$((http_port + 363))` is a magic number. For the default port 8080, this gives 8443, which is conventional. But for any other port, the result is arbitrary (e.g., port 9080 yields HTTPS port 9443, but port 3000 yields 3363, which is unexpected). The `TEST_DASHBOARD_PORT=$((http_port + 2))` is similarly fragile -- port 8080 gives 8082, but port 3000 gives 3002.

**12. `cmd_stop` deletes `.generated/` but fallback `docker compose down` may not work.**
File: `/home/i9user/Projects/DS/dev-strap/devstack.sh`, lines 211-213.
If the user runs `docker compose stop` manually (outside devstack.sh), the containers will stop but `.generated/` and volumes remain. Then running `./devstack.sh stop` will find the compose file and attempt a proper teardown. But if `.generated/` was manually deleted, the fallback `docker compose -p "${PROJECT_NAME}" down -v` may not find the containers since they were started with `-f` pointing to a specific compose file. This is an edge case but could leave orphaned containers and volumes.

**13. `cmd_mocks` suggests `./devstack.sh restart` but that command does `stop` (with volume deletion) then `start`.**
File: `/home/i9user/Projects/DS/dev-strap/devstack.sh`, line 405.
The `cmd_mocks` output says "Run './devstack.sh restart'" but `cmd_restart` calls `cmd_stop` (which runs `docker compose down -v`, deleting all volumes including database data) then `cmd_start`. This means "restart" is a destructive operation that drops the database. The docs should call this out, or the `restart` command should offer a non-destructive option.

**14. `verify-mocks` uses `--no-check-certificate` and tests only root path.**
File: `/home/i9user/Projects/DS/dev-strap/devstack.sh`, lines 742-747.
The verification sends `wget` to `https://${domain}/` with `--no-check-certificate`. This proves DNS resolution and Caddy routing work, but does NOT verify that (a) the app trusts the CA certificate, or (b) WireMock has any actual mappings loaded. A 404 from WireMock is treated as PASS ("routed to WireMock (404: no mapping for /)"), which is technically correct but could mask configuration problems.

**15. Swagger UI template mounts a hardcoded file path that likely does not exist.**
File: `/home/i9user/Projects/DS/dev-strap/templates/extras/swagger-ui/service.yml`, line 9.
The template mounts `${APP_SOURCE}/docs/openapi.json:/spec/openapi.json:ro`. The example app and scaffolded apps do not have a `docs/openapi.json` file. Docker will create an empty directory at the mount point if the source does not exist, causing Swagger UI to fail with a confusing error.

## Documentation Accuracy

### Where docs say one thing but the system does another

1. **QUICKSTART.md, lines 48-51:** Documents `./devstack.sh init --preset full-stack` (and three other presets). The `cmd_init` function has no `--preset` support. These commands do not work.

2. **QUICKSTART.md, line 62:** Says the wizard walks through "frontend (Vite), database, services (Redis, Mailpit, NATS, MinIO), tooling, and observability." The actual `cmd_init` function (lines 792-814) only prompts for project name, app type, database, extras (just "redis, mailpit"), and HTTP port. No frontend, no tooling, no observability prompts.

3. **README.md, line 41:** Says `./devstack.sh test` should show "6/6 pass." The test suite has exactly 6 tests across 4 spec files, so this claim is likely accurate for the example app.

4. **README.md, line 73 (app/src/index.js, line 73):** The example app's HTML mentions "nginx" instead of "Caddy."

5. **ARCHITECTURE.md, lines 204-205:** Describes path-based routing as `/api/* -> app (backend)` but does not mention that `handle_path` strips the prefix. A developer reading this would expect their `/api/users` route to work, but it would need to be just `/users`.

6. **README.md, line 172:** Lists WireMock's port as 8443 in the catalog table. WireMock runs on port 8080 internally. Port 8443 is the HTTPS port exposed to the host for the Caddy proxy, not a direct WireMock port. This conflation is misleading.

7. **docs/ADDING_MOCKS.md, line 9:** Shows `./devstack.sh stop && ./devstack.sh start` for applying new mocks. The `stop` command deletes all volumes including the database. A new mock (adding a new domain) requires a restart because cert-gen and DNS aliases need to be rebuilt, but it does not require losing the database. There is no non-destructive restart option.

### What's missing from documentation

- No documentation for the `handle_path` path-stripping behavior and its implications for backend route definitions in frontend+backend mode.
- No documentation for the `record` command's actual workflow from the app's perspective (how does the app route traffic through the recorder container?).
- No documentation for what happens when `project.env` variables are missing or have incorrect values.
- No migration guide from the previous nginx-based system (the Caddyfile generator still says "Replaces: core/nginx/generate-conf.sh" in its header).

## Architecture Critique

### Strengths

1. **Convention over configuration.** The `mocks/` directory convention is excellent -- `ls mocks/` tells you what's mocked, and the filesystem structure maps directly to the generated infrastructure. This is the right abstraction for a development tool.

2. **Clean separation between source-of-truth and derived.** The `.generated/` directory with gitignored contents makes it obvious what's derived and expendable. The `.gitignore` correctly lists `.generated/` and `tests/results/`.

3. **Single-script CLI.** Having everything in one `devstack.sh` file makes the tool portable and eliminates build steps. The script is well-organized with clear function separation.

### Weaknesses

1. **sed-based templating is fragile.** The compose generator uses 15+ chained `sed` commands for variable substitution (`core/compose/generate.sh`, lines 103-120). This breaks silently if a variable value contains pipe characters (`|`, used as the sed delimiter), special regex characters, or newlines. A more robust approach would use envsubst or a proper template engine.

2. **Monolithic 1688-line bash script.** `devstack.sh` contains the CLI, lifecycle management, mock management, test runner, project scaffolding, AND the entire contract validation/generation system (including a 150-line jq expression). This makes it hard to test individual functions and easy to introduce regressions. The contract code alone (~650 lines) should be in a separate file.

3. **No validation of `project.env` values.** `load_config` does `source "${DEVSTACK_DIR}/project.env"` with no validation. If `APP_TYPE` is set to a nonexistent template, the compose generator produces an error message but only after building the cert-gen and web service blocks. If `DB_TYPE` is set to an unsupported value, the `case` statement silently defaults `DB_PORT` to 3306. If `NETWORK_SUBNET` is malformed, Docker will fail at container start with an unhelpful error.

4. **`stop` is always destructive.** `cmd_stop` always runs `docker compose down -v` and `rm -rf "${GENERATED_DIR}"`. There is no way to stop containers without deleting volumes (database data). The `restart` command inherits this behavior, making it destructive by default. This is the right choice for "clean slate" semantics but surprising for users who want to restart after a config change without losing their database.

5. **cert-gen runs every time but caches.** The cert-gen container starts on every `start` but checks for existing certs and skips if found (line 22-26 of `core/certs/generate.sh`). However, if the user adds a new mock domain, the existing cert won't include the new domain's SAN. There is no detection for "domains changed, need to regenerate." The user must `FORCE_REGEN=1` or do a full `stop`/`start` (which deletes the cert volume). This is not documented.

6. **Absolute paths in generated compose.** The generated `docker-compose.yml` uses absolute host paths (e.g., `/home/user/projects/dev-strap/app:/app`). This makes the compose file non-portable across machines and breaks in CI environments where the workspace path differs. The TODO.md (line 19) acknowledges this as a known issue.

7. **No input sanitization on mock domain names.** The domain parsing in all three generators (`core/caddy/generate-caddyfile.sh`, `core/compose/generate.sh`, `core/certs/generate.sh`) strips whitespace and skips comments, but does not validate that the domain is a valid hostname. A domain containing spaces (after stripping), special characters, or wildcards could produce malformed Caddyfiles, docker-compose YAML, or certificate SANs.

## Recommendations (ordered by priority)

### Fix immediately

1. **Wire up `EXTRAS_DEPENDS` in the compose generator** (or remove it). The variable is built but never used. The app service should `depends_on` extras with `condition: service_healthy` so Redis/Mailpit/etc. are ready before the app starts. File: `core/compose/generate.sh`.

2. **Fix PHP-Laravel `DB_CONNECTION` to vary with `DB_TYPE`.** Add a `DB_CONNECTION` variable derived from `DB_TYPE` (mysql for mariadb, pgsql for postgres) and substitute it into the PHP template. File: `templates/apps/php-laravel/service.yml`, `core/compose/generate.sh`.

3. **Fix the nginx reference in the example app.** Change "nginx" to "Caddy" on line 73 of `app/src/index.js`.

4. **Remove or implement `init --preset`.** Either add `--preset` flag parsing to `cmd_init()` or remove all references from QUICKSTART.md and README.md. Currently the docs describe a feature that does not exist.

### Fix soon

5. **Add healthchecks to app templates.** Each app template's `service.yml` should define a healthcheck (e.g., `wget --spider http://localhost:3000/health`), and the web service should depend on `app: condition: service_healthy` instead of `service_started`.

6. **Document `handle_path` prefix stripping.** Add a clear warning in ARCHITECTURE.md and PROJECT_SETUP.md that in frontend+backend mode, backend routes should NOT include the `/api` prefix because Caddy strips it. Alternatively, switch to `handle` with a path matcher instead of `handle_path` if the prefix should be preserved.

7. **Add cert domain change detection.** Before skipping cert generation, compare the current `domains.txt` with the SANs in the existing cert. Regenerate if they differ. This would make adding new mocks less error-prone.

8. **Fix Swagger UI template's hardcoded file mount.** Either make the mount conditional on the file existing, or document that users must create `app/docs/openapi.json` before enabling swagger-ui.

### Fix eventually

9. **Add a non-destructive restart.** Implement `./devstack.sh restart --keep-volumes` (or make that the default and add `restart --clean`). The current behavior of always deleting database volumes on restart is hostile to iterative development.

10. **Split `devstack.sh`.** Extract the contract/PowerHouse code (~650 lines) into `core/contract/bootstrap.sh` or similar. The main script would `source` it. This improves readability and testability.

11. **Replace sed templating with envsubst.** Use `envsubst` (available in most Docker images and easily installable) for variable substitution in templates. This eliminates the fragile 15-way sed chain and the risk of special character injection.

12. **Add `project.env` validation.** After sourcing, verify that critical variables (`PROJECT_NAME`, `APP_TYPE`, `APP_SOURCE`) are set and valid. Check `APP_TYPE` against available templates. Validate `NETWORK_SUBNET` format. Fail fast with clear messages.

### Fine as-is

- The WireMock mapping path collision issue (finding #8) is a real concern but is inherent to WireMock's design. Documenting it is sufficient; building collision detection is low ROI.
- The init port arithmetic (finding #11) is unconventional but harmless for the common case.
- The `verify-mocks` 404-as-pass behavior (finding #14) is a reasonable tradeoff for a quick smoke test.
