## 1. Non-Destructive Restart

### Current behavior

`cmd_stop` (line 204) runs:
```bash
docker compose -f ... -p "${PROJECT_NAME}" down -v --remove-orphans
```

The `-v` flag removes **all** named volumes: database data, cert volume, go-modules, cargo-target, python-cache. `cmd_restart` (line 411) is literally `cmd_stop` followed by `cmd_start`, so every restart is a full teardown.

### What should survive a restart

| Resource                | Default on restart | Rationale                                              |
|-------------------------|-------------------|--------------------------------------------------------|
| `.generated/`           | Delete            | Regenerated by `cmd_generate` inside `cmd_start`       |
| Database volumes        | **Preserve**      | Data seeded by migrations/init scripts; loss is costly |
| Module caches           | **Preserve**      | `go-modules`, `cargo-registry`, `cargo-target`, `python-cache`, `node_modules` anonymous volume |
| Cert volume             | **Preserve***     | Only regenerate if domains changed (see section 2)     |
| Test results            | Preserve          | User reviews after test runs                           |
| Containers + networks   | Delete            | Recreated from regenerated compose                     |

*Cert volume preservation is conditional -- see section 2 for detection logic.

### Options analysis

**Option A: `restart` preserves volumes, `restart --clean` deletes them**
- Pros: Non-breaking for `restart` muscle memory. Clean option explicit.
- Cons: `stop` still has the question: should it delete volumes?

**Option B: `restart` preserves volumes, `stop` preserves volumes, new `clean` command deletes everything**
- Pros: Clean separation of concerns. Three distinct actions.
- Cons: Users accustomed to `stop` being a full teardown will accumulate orphan volumes.

**Option C: `stop` asks whether to delete volumes (interactive)**
- Rejected: Breaks scripted/CI usage. Interactive prompts in lifecycle commands are an anti-pattern.

**Option D: `stop --keep-data` / `stop --clean` flags**
- Pros: Explicit, scriptable.
- Cons: Need to choose a default. If `stop` defaults to keeping data, silent accumulation. If it defaults to deleting, same problem as today.

### Recommendation: Option A + modified stop

Adopt a two-tier model:

| Command                    | Containers | Networks | `.generated/` | Named Volumes | Root-owned files |
|----------------------------|:----------:|:--------:|:--------------:|:-------------:|:----------------:|
| `restart`                  | Recreate   | Recreate | Regenerate     | Keep          | Keep             |
| `restart --clean`          | Recreate   | Recreate | Regenerate     | Delete        | Delete           |
| `stop`                     | Remove     | Remove   | Delete         | Keep          | Keep             |
| `stop --clean`             | Remove     | Remove   | Delete         | Delete        | Delete           |

### Implementation sketch

```bash
cmd_stop() {
    local clean=0
    while [[ $# -gt 0 ]]; do
        case "$1" in
            --clean) clean=1; shift ;;
            *) log_err "Unknown flag: $1"; exit 1 ;;
        esac
    done

    log "Stopping DevStack..."

    if [ -f "${GENERATED_DIR}/docker-compose.yml" ]; then
        if [ "${clean}" = "1" ]; then
            docker compose -f "${GENERATED_DIR}/docker-compose.yml" \
                -p "${PROJECT_NAME}" down -v --remove-orphans 2>/dev/null || true
        else
            docker compose -f "${GENERATED_DIR}/docker-compose.yml" \
                -p "${PROJECT_NAME}" down --remove-orphans 2>/dev/null || true
        fi
    else
        if [ "${clean}" = "1" ]; then
            docker compose -p "${PROJECT_NAME}" down -v --remove-orphans 2>/dev/null || true
        else
            docker compose -p "${PROJECT_NAME}" down --remove-orphans 2>/dev/null || true
        fi
    fi

    rm -rf "${GENERATED_DIR}"

    # Only clean root-owned test artifacts on --clean
    if [ "${clean}" = "1" ]; then
        if [ -d "${DEVSTACK_DIR}/tests/results" ] || \
           [ -d "${DEVSTACK_DIR}/tests/playwright/node_modules" ]; then
            docker run --rm \
                -v "${DEVSTACK_DIR}/tests:/data" \
                alpine sh -c "rm -rf /data/results/* /data/playwright/node_modules \
                    /data/playwright/package-lock.json" 2>/dev/null || true
        fi
        # Clean recordings (section 4)
        _clean_recordings
    fi

    if [ "${clean}" = "1" ]; then
        log_ok "DevStack stopped. All containers, volumes, and generated files removed."
    else
        log_ok "DevStack stopped. Volumes preserved."
        log "Run './devstack.sh stop --clean' to also remove database and cache volumes."
    fi
}

cmd_restart() {
    local args=("$@")
    cmd_stop "${args[@]}"
    cmd_start
}
```

### Docker Compose commands reference

The implementation relies on two `docker compose down` variants:

- `docker compose down --remove-orphans` -- removes containers and networks, keeps named volumes
- `docker compose down -v --remove-orphans` -- removes containers, networks, AND all named volumes

Using `docker compose up -d --force-recreate` (which recreates containers from updated config while keeping volumes) was considered for `restart`, but it doesn't pick up changes to the compose file itself (new services, changed volumes). Since `cmd_start` regenerates the compose file first, the sequence must be: stop containers, regenerate config, start fresh. The `down` (no `-v`) + `up -d` path achieves this.

### Migration concern

Current users expect `stop` to delete everything. The log message on `stop` (without `--clean`) should explicitly say volumes are preserved and tell them how to fully clean up. The `restart` behavior change is purely beneficial -- nobody wants to lose their database on restart.

## 2. Cert Domain Change Detection

### Current behavior

`core/certs/generate.sh` (line 22):
```bash
if [ -f "${CERT_DIR}/server.crt" ] && [ "${FORCE_REGEN:-0}" != "1" ]; then
    echo "[cert-gen] Certificates already exist. Skipping generation."
    exit 0
fi
```

If the cert volume survives (as proposed in section 1), and a user adds a new mock domain, the old cert will be missing the new SAN. HTTPS handshakes to the new domain will fail with certificate errors.

### Detection approach

Before skipping, compare the SANs in the existing cert against the current `domains.txt`. If they differ, regenerate.

This must run inside the cert-gen container because it has `openssl` (installed by the entrypoint `apk add openssl`).

### Implementation sketch

Replace the early-exit block in `core/certs/generate.sh`:

```bash
# If certs already exist, check if domains have changed
if [ -f "${CERT_DIR}/server.crt" ] && [ "${FORCE_REGEN:-0}" != "1" ]; then
    # Extract current SANs from the existing cert
    CURRENT_SANS=$(openssl x509 -in "${CERT_DIR}/server.crt" -noout -ext subjectAltName 2>/dev/null \
        | grep -oP 'DNS:\K[^,\s]+' \
        | sort)

    # Build expected SANs from domains.txt + defaults
    EXPECTED_SANS="localhost"
    if [ -f "${DOMAINS_FILE}" ]; then
        while IFS= read -r domain || [ -n "${domain}" ]; do
            domain=$(echo "${domain}" | tr -d '[:space:]')
            [ -z "${domain}" ] && continue
            [[ "${domain}" == \#* ]] && continue
            EXPECTED_SANS="${EXPECTED_SANS}"$'\n'"${domain}"
        done < "${DOMAINS_FILE}"
    fi
    EXPECTED_SANS="${EXPECTED_SANS}"$'\n'"${PROJECT_NAME:-devstack}.local"
    EXPECTED_SANS=$(echo "${EXPECTED_SANS}" | sort)

    if [ "${CURRENT_SANS}" = "${EXPECTED_SANS}" ]; then
        echo "[cert-gen] Certificates exist and SANs match. Skipping generation."
        exit 0
    else
        echo "[cert-gen] Domain change detected. Regenerating certificates..."
        echo "[cert-gen] Current SANs: $(echo "${CURRENT_SANS}" | tr '\n' ', ')"
        echo "[cert-gen] Expected SANs: $(echo "${EXPECTED_SANS}" | tr '\n' ', ')"
    fi
fi
```

### Edge cases

1. **First run (no cert exists)**: The `if [ -f server.crt ]` check fails, falls through to generation. No change needed.
2. **Domain removed**: The comparison catches this too -- the cert has extra SANs. This is technically harmless (extra SANs don't break anything), but regenerating keeps the cert clean. Recommend regenerating on any difference.
3. **`FORCE_REGEN=1`**: Bypasses the check entirely, as before.
4. **Alpine `grep -P` availability**: Alpine's `grep` may lack `-P` (Perl regex). Use `sed` or `awk` instead:
   ```bash
   CURRENT_SANS=$(openssl x509 -in "${CERT_DIR}/server.crt" -noout -ext subjectAltName 2>/dev/null \
       | tr ',' '\n' \
       | sed -n 's/.*DNS:\([^ ]*\)/\1/p' \
       | sort)
   ```
5. **IP SANs**: The comparison only checks DNS SANs. IP SANs (127.0.0.1) are static and never change, so ignoring them in the comparison is correct.

### Performance

The `openssl x509 -noout -ext` command is near-instant. The sorted-string comparison adds negligible overhead to container startup. This runs only once per `start`, inside the cert-gen container.

## 3. project.env Validation

### Current behavior

`load_config()` (line 52) does:
```bash
source "${DEVSTACK_DIR}/project.env"
```

No validation. If `APP_TYPE=rails` (nonexistent template), the error surfaces much later as "No app template found" during compose generation. If `NETWORK_SUBNET` is malformed, Docker fails with a cryptic error at `up` time.

### Validation function design

```bash
validate_config() {
    local errors=()

    # --- Required variables ---
    [ -z "${PROJECT_NAME:-}" ] && errors+=("PROJECT_NAME is required")
    [ -z "${APP_TYPE:-}" ]     && errors+=("APP_TYPE is required")
    [ -z "${APP_SOURCE:-}" ]   && errors+=("APP_SOURCE is required")

    # --- PROJECT_NAME format ---
    # Must start with lowercase letter, contain only lowercase, digits, hyphens
    # Same rule as the contract validation
    if [ -n "${PROJECT_NAME:-}" ]; then
        if ! echo "${PROJECT_NAME}" | grep -qE '^[a-z][a-z0-9-]*$'; then
            errors+=("PROJECT_NAME '${PROJECT_NAME}' is invalid (must match [a-z][a-z0-9-]*)")
        fi
    fi

    # --- APP_TYPE must match a template directory ---
    if [ -n "${APP_TYPE:-}" ]; then
        if [ ! -d "${DEVSTACK_DIR}/templates/apps/${APP_TYPE}" ]; then
            local available
            available=$(ls -1 "${DEVSTACK_DIR}/templates/apps/" 2>/dev/null | tr '\n' ', ' | sed 's/,$//')
            errors+=("APP_TYPE '${APP_TYPE}' not found. Available: ${available}")
        fi
    fi

    # --- DB_TYPE must match a template directory or be "none" ---
    if [ -n "${DB_TYPE:-}" ] && [ "${DB_TYPE}" != "none" ]; then
        if [ ! -d "${DEVSTACK_DIR}/templates/databases/${DB_TYPE}" ]; then
            local available
            available=$(ls -1 "${DEVSTACK_DIR}/templates/databases/" 2>/dev/null | tr '\n' ', ' | sed 's/,$//')
            errors+=("DB_TYPE '${DB_TYPE}' not found. Available: ${available}, none")
        fi
    fi

    # --- FRONTEND_TYPE must match a template directory or be "none"/empty ---
    if [ -n "${FRONTEND_TYPE:-}" ] && [ "${FRONTEND_TYPE}" != "none" ]; then
        if [ ! -d "${DEVSTACK_DIR}/templates/frontends/${FRONTEND_TYPE}" ]; then
            local available
            available=$(ls -1 "${DEVSTACK_DIR}/templates/frontends/" 2>/dev/null | tr '\n' ', ' | sed 's/,$//')
            errors+=("FRONTEND_TYPE '${FRONTEND_TYPE}' not found. Available: ${available}, none")
        fi
    fi

    # --- EXTRAS: each entry must match a template directory ---
    if [ -n "${EXTRAS:-}" ]; then
        IFS=',' read -ra extra_list <<< "${EXTRAS}"
        for extra in "${extra_list[@]}"; do
            extra=$(echo "${extra}" | tr -d '[:space:]')
            [ -z "${extra}" ] && continue
            if [ ! -d "${DEVSTACK_DIR}/templates/extras/${extra}" ]; then
                local available
                available=$(ls -1 "${DEVSTACK_DIR}/templates/extras/" 2>/dev/null \
                    | tr '\n' ', ' | sed 's/,$//')
                errors+=("EXTRAS entry '${extra}' not found. Available: ${available}")
            fi
        done
    fi

    # --- NETWORK_SUBNET basic format check ---
    if [ -n "${NETWORK_SUBNET:-}" ]; then
        if ! echo "${NETWORK_SUBNET}" | grep -qE '^[0-9]+\.[0-9]+\.[0-9]+\.[0-9]+/[0-9]+$'; then
            errors+=("NETWORK_SUBNET '${NETWORK_SUBNET}' is not valid CIDR (expected x.x.x.x/n)")
        fi
    fi

    # --- Ports: must be numeric and in valid range ---
    local port_vars=(HTTP_PORT HTTPS_PORT TEST_DASHBOARD_PORT MAILPIT_PORT)
    for var in "${port_vars[@]}"; do
        local val="${!var:-}"
        if [ -n "${val}" ]; then
            if ! echo "${val}" | grep -qE '^[0-9]+$'; then
                errors+=("${var} '${val}' must be a number")
            elif [ "${val}" -lt 1 ] || [ "${val}" -gt 65535 ]; then
                errors+=("${var} '${val}' must be between 1 and 65535")
            fi
        fi
    done

    # --- Report errors ---
    if [ ${#errors[@]} -gt 0 ]; then
        log_err "project.env validation failed:"
        for err in "${errors[@]}"; do
            log_err "  - ${err}"
        done
        exit 1
    fi
}
```

### When to run validation

Validation must run after `source project.env` and before any generation or Docker commands:

- `cmd_start` -- yes, first thing after `load_config`
- `cmd_restart` -- yes, inherits from `cmd_start`
- `cmd_generate` -- yes, catches problems before writing files
- `cmd_stop` -- no, stop should always work even with invalid config (needs PROJECT_NAME only)
- `cmd_init` -- no, it creates project.env, not reads it
- Contract commands (`--options`, `--bootstrap`) -- separate validation exists already

### Integration point

Modify `load_config()`:
```bash
load_config() {
    if [ ! -f "${DEVSTACK_DIR}/project.env" ]; then
        log_err "project.env not found in ${DEVSTACK_DIR}"
        exit 1
    fi
    source "${DEVSTACK_DIR}/project.env"
    validate_config
}
```

For `cmd_stop`, use a minimal loader that only sources without validation:
```bash
load_config_minimal() {
    if [ ! -f "${DEVSTACK_DIR}/project.env" ]; then
        log_err "project.env not found in ${DEVSTACK_DIR}"
        exit 1
    fi
    source "${DEVSTACK_DIR}/project.env"
}
```

### Warning vs. error

All checks above are hard errors (exit 1). For a future iteration, warnings could be added for:
- `APP_INIT_SCRIPT` pointing to a nonexistent file (currently silently skipped)
- Ports below 1024 (require root or `CAP_NET_BIND_SERVICE`)
- `EXTRAS` containing `db-ui` when `DB_TYPE=none` (B6 guard)

## 4. Recordings Cleanup

### Current behavior

`cmd_stop` (line 219-224) cleans:
```bash
docker run --rm \
    -v "${DEVSTACK_DIR}/tests:/data" \
    alpine sh -c "rm -rf /data/results/* /data/playwright/node_modules \
        /data/playwright/package-lock.json"
```

This misses `mocks/*/recordings/` directories, which are created by `cmd_record` (line 562-568) and are root-owned because WireMock writes them from inside a container.

### Recording lifecycle

1. `cmd_record stripe` creates `mocks/stripe/recordings/{mappings,__files}/` and runs a WireMock recorder container that writes root-owned JSON files there.
2. User reviews the recordings.
3. `cmd_apply_recording stripe` copies recordings into `mocks/stripe/mappings/`, then deletes `mocks/stripe/recordings/`.

The problem: if the user runs `record` but never runs `apply-recording`, root-owned recordings persist. `cmd_stop` does not clean them.

### Design

Recordings are intermediate artifacts that the user might want to review before applying. They should NOT be deleted on regular `stop` (user may stop and start without losing recorded data they haven't applied yet). They SHOULD be deleted on `stop --clean`.

```bash
_clean_recordings() {
    local has_recordings=0
    for rec_dir in "${DEVSTACK_DIR}"/mocks/*/recordings; do
        [ -d "${rec_dir}" ] || continue
        has_recordings=1
        docker run --rm \
            -v "${rec_dir}:/data" \
            alpine rm -rf /data/mappings /data/__files 2>/dev/null || true
        rmdir "${rec_dir}" 2>/dev/null || true
    done
    if [ "${has_recordings}" = "1" ]; then
        log "Cleaned up recording directories"
    fi
}
```

This function is called from `cmd_stop --clean` (see section 1 sketch).

### Alternative: clean recordings on regular `stop` with a warning

If the team prefers always cleaning recordings on stop, modify the stop message:
```
[devstack] Warning: Unapplied recordings in mocks/stripe/recordings/ were deleted.
[devstack] Use 'apply-recording stripe' before stopping next time.
```

Recommendation: Keep recordings on regular `stop`. Delete on `stop --clean`. The user explicitly opted for a clean slate.

## 5. Per-Language App Healthchecks

### Current state

None of the five app templates (`node-express`, `php-laravel`, `go`, `python-fastapi`, `rust`) have a `healthcheck` block. The web service (`core/compose/generate.sh` line 303) uses:
```yaml
app:
  condition: service_started
```

This means Caddy starts proxying as soon as the app container exists, not when the app is listening. During cold starts (dependency installation, compilation), requests hit Caddy before the app is ready, producing 502 errors.

### Healthcheck approach

Use TCP connection check rather than HTTP response check. Reasons:

1. The app may not have a `/` route returning 200 (user apps vary).
2. A dedicated `/health` endpoint would require adding code to every scaffolded app template, increasing the maintenance burden on all five Dockerfiles.
3. TCP check (`wget --spider`) verifies "the process is listening on the port" which is sufficient for the depends_on gate -- once the port is open, the app is ready to handle requests.

### Per-language specifications

All templates except PHP-FPM expose an HTTP port (3000). PHP-FPM uses FastCGI on port 9000.

#### Node.js (`templates/apps/node-express/service.yml`)

```yaml
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:3000/"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```

`start_period: 10s` covers `npm install` on first start (node_modules anonymous volume is empty). Retries set to 30 (30 * 5s = 150s max wait) because dependency installation can be slow.

#### Go (`templates/apps/go/service.yml`)

```yaml
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:3000/"]
      interval: 5s
      timeout: 3s
      retries: 60
      start_period: 15s
```

Go requires compilation on first start. With Air (live reload), the first build can take 30-60 seconds. Retries set to 60 (60 * 5s = 300s max wait). `start_period` of 15s gives the build time before health failures count.

#### Python FastAPI (`templates/apps/python-fastapi/service.yml`)

```yaml
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:3000/"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```

uvicorn starts fast once dependencies are installed. Similar profile to Node.

#### Rust (`templates/apps/rust/service.yml`)

```yaml
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:3000/"]
      interval: 5s
      timeout: 3s
      retries: 120
      start_period: 30s
```

Rust compilation is the slowest of all languages. First build with dependencies can take several minutes. Retries set to 120 (120 * 5s = 600s = 10 min max wait). `start_period: 30s` is conservative.

#### PHP-Laravel (`templates/apps/php-laravel/service.yml`)

PHP-FPM does not serve HTTP. It speaks FastCGI on port 9000. Two options:

**Option 1: `cgi-fcgi` probe**
```yaml
    healthcheck:
      test: ["CMD-SHELL", "SCRIPT_NAME=/ping SCRIPT_FILENAME=/ping REQUEST_METHOD=GET cgi-fcgi -bind -connect 127.0.0.1:9000 || exit 1"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```
Requires `libfcgi-bin` installed in the container (provides `cgi-fcgi`). The PHP-FPM pool must have `pm.status_path = /ping` or `ping.path = /ping` configured.

**Option 2: Check PID file or process**
```yaml
    healthcheck:
      test: ["CMD-SHELL", "test -f /run/php-fpm.pid || exit 1"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```
Lightweight but only confirms the master process started, not that workers are ready.

**Option 3: PHP-FPM built-in ping**
Configure in `www.conf` (or mount a config snippet):
```ini
ping.path = /health
ping.response = ok
```
Then:
```yaml
    healthcheck:
      test: ["CMD-SHELL", "cgi-fcgi -bind -connect 127.0.0.1:9000 /health || exit 1"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```

**Recommendation for PHP**: Use `php-fpm -t` as the simplest reliable check. It validates the FPM configuration is loaded and the master process is running. No additional packages needed:
```yaml
    healthcheck:
      test: ["CMD-SHELL", "php-fpm -t 2>/dev/null || exit 1"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```

Note: `php-fpm -t` only validates config, not that workers are accepting connections. For true readiness, install `libfcgi-bin` in the Dockerfile and use the `cgi-fcgi` approach. Given this is a dev environment (not production with traffic), `php-fpm -t` is sufficient.

#### Vite frontend (`templates/frontends/vite/service.yml`)

```yaml
    healthcheck:
      test: ["CMD", "wget", "--spider", "-q", "http://localhost:5173/"]
      interval: 5s
      timeout: 3s
      retries: 30
      start_period: 10s
```

Also requires updating `core/compose/generate.sh` -- the web service's `depends_on` for frontend (line 293) currently uses `service_started`:
```yaml
      frontend:
        condition: service_started
```
Change to `service_healthy`.

### Compose generator changes

In `core/compose/generate.sh`, update the web service `depends_on` block.

Line 303 currently:
```yaml
      app:
        condition: service_started
```

Change to:
```yaml

```

And line 293 for frontend:
```yaml

```

```yaml

        condition: service_healthy
```

### `wget` availability in containers

- **Node.js** (based on `node:` image): Has `wget` (Debian-based).
- **Go** (typically `golang:` image): Has `wget` (Debian-based).
- **Python** (typically `python:` image): Has `wget` (Debian-based).
- **Rust** (typically `rust:` image): Has `wget` (Debian-based).
- **PHP-FPM** (`php:8.3-fpm`): Has `wget` (Debian-based). But FPM doesn't serve HTTP, so wget to localhost:9000 won't work anyway.
- **Vite** (typically `node:` image): Has `wget`.

All app images are Debian-based, so `wget` is available. No additional packages needed for the HTTP-based healthchecks.

## 6. Init Command Expansion

### Current state

`cmd_init` (line 779) hardcodes three app types in the prompt:
```
App type (node-express, php-laravel, go) [node-express]:
```

The actual filesystem has five: `node-express`, `php-laravel`, `go`, `python-fastapi`, `rust`. No frontend prompt exists. No observability/tooling prompts. The `--preset` flag is documented in `docs/QUICKSTART.md` (lines 43-50) and defined in `contract/manifest.json` but not implemented.

### Design: filesystem-driven prompts

```bash
cmd_init() {
    local preset=""

    # Parse flags
    while [[ $# -gt 0 ]]; do
        case "$1" in
            --preset)
                preset="$2"
                shift 2
                ;;
            *)
                log_err "Unknown flag: $1"
                exit 1
                ;;
        esac
    done

    # --- Preset path ---
    if [ -n "${preset}" ]; then
        _init_from_preset "${preset}"
        return
    fi

    # --- Interactive path ---
    echo ""
    log "DevStack Project Setup"
    echo ""

    # Check if already configured (existing logic)
    ...

    # Project name
    echo -n "  Project name [myproject]: "
    read -r input_name
    local proj_name="${input_name:-myproject}"

    # App type -- read from filesystem
    local app_types
    app_types=$(ls -1 "${DEVSTACK_DIR}/templates/apps/" | tr '\n' ', ' | sed 's/,$//')
    echo -n "  App type (${app_types}) [node-express]: "
    read -r input_type
    local app_type="${input_type:-node-express}"

    # Validate
    if [ ! -d "${DEVSTACK_DIR}/templates/apps/${app_type}" ]; then
        log_err "Unknown app type '${app_type}'. Available:"
        ls -1 "${DEVSTACK_DIR}/templates/apps/"
        exit 1
    fi

    # Database -- read from filesystem
    local db_types
    db_types=$(ls -1 "${DEVSTACK_DIR}/templates/databases/" | tr '\n' ', ' | sed 's/,$//')
    echo -n "  Database (${db_types}, none) [mariadb]: "
    read -r input_db
    local db_type="${input_db:-mariadb}"

    # Frontend -- read from filesystem
    local frontend_types
    frontend_types=$(ls -1 "${DEVSTACK_DIR}/templates/frontends/" 2>/dev/null \
        | tr '\n' ', ' | sed 's/,$//')
    if [ -n "${frontend_types}" ]; then
        echo -n "  Frontend (${frontend_types}, none) [none]: "
```

```bash
_init_from_preset() {
    local preset_name="$1"
    local manifest="${DEVSTACK_DIR}/contract/manifest.json"

    if ! command -v jq &>/dev/null; then
        log_err "jq is required for --preset. Install: https://jqlang.github.io/jq/download/"
        exit 1
    fi

    if [ ! -f "${manifest}" ]; then
        log_err "Manifest not found at ${manifest}"
        exit 1
    fi

    # Check preset exists
    local preset_data
    preset_data=$(jq -r --arg p "${preset_name}" '.presets[$p] // empty' "${manifest}")
    if [ -z "${preset_data}" ]; then
        local available
        available=$(jq -r '.presets | keys | join(", ")' "${manifest}")
        log_err "Unknown preset '${preset_name}'. Available: ${available}"
        exit 1
    fi

    log "Initializing from preset: ${preset_name}"
    log "$(jq -r --arg p "${preset_name}" '.presets[$p].description' "${manifest}")"

    # Extract selections
    local app_type db_type frontend_type extras

    # Check if preset pre-selects the app, or if it needs prompting
    app_type=$(jq -r --arg p "${preset_name}" \
        '.presets[$p].selections.app[0] // empty' "${manifest}")

    if [ -z "${app_type}" ]; then
        # Preset has "prompts": ["app"] -- ask the user
        local app_types
        app_types=$(ls -1 "${DEVSTACK_DIR}/templates/apps/" | tr '\n' ', ' | sed 's/,$//')
        echo -n "  App type (${app_types}) [node-express]: "
        read -r input_type
        app_type="${input_type:-node-express}"
    fi

    # Extract other selections from preset
    db_type=$(jq -r --arg p "${preset_name}" \
        '.presets[$p].selections.database[0] // "none"' "${manifest}")
    frontend_type=$(jq -r --arg p "${preset_name}" \
        '.presets[$p].selections.frontend[0] // "none"' "${manifest}")

    # Build extras list from services + tooling + observability selections
    extras=$(jq -r --arg p "${preset_name}" '
        [
            (.presets[$p].selections.services // []),
            (.presets[$p].selections.tooling // []),
            (.presets[$p].selections.observability // [])
        ] | flatten | join(",")
    ' "${manifest}")

    # Prompt for project name
    echo -n "  Project name [myproject]: "
    read -r input_name
    local proj_name="${input_name:-myproject}"

    # Write project.env with preset selections
    # (same template as interactive path)
    ...

    log_ok "Initialized from preset '${preset_name}'"
}
```

### What NOT to add to init

Keep `init` simple. Do not add:
- Mock service scaffolding (use `new-mock` separately)
- App code scaffolding beyond the Dockerfile copy (user brings their own app)
- Detailed port configuration (defaults are fine; edit project.env to customize)

The `--bootstrap` contract path (PowerHouse integration) already handles the complex case with full wiring, overrides, and validation. `init` is for humans typing at a terminal.

### Implementation order

Recommended sequence to minimize risk and dependency chains:

1. **App healthchecks** (B1) -- pure template additions, no logic changes. Unblocks everything that depends on `service_healthy`.
2. **project.env validation** (D5) -- catches errors early, independent of other changes.
3. **Non-destructive restart** (D1) -- changes `cmd_stop` and `cmd_restart`. Must be done before cert detection matters.
4. **Cert domain change detection** (D2) -- only matters once volumes survive restart.
5. **Recordings cleanup** (D6) -- hooks into the `--clean` flag from step 3.
6. **Init expansion** (B2) -- UI improvement, independent of the above but benefits from validation (step 2).
