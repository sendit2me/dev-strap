# AI Agent Bootstrap

## What this project is

A Docker-based development environment with transparent HTTPS mock interception. Your app makes requests to external APIs exactly as it would in production. Docker DNS, Caddy (reverse proxy), and WireMock intercept those requests and return stub responses. No code changes required.

## What to edit vs what is generated

**Source files (edit these):**

| File | Purpose |
|------|---------|
| `project.env` | Central configuration (ports, names, credentials) |
| `services/app.yml` | Application service definition |
| `services/database.yml` | Database service definition |
| `services/redis.yml`, etc. | Other infrastructure services |
| `services/cert-gen.yml` | Certificate generator service |
| `mocks/*/domains` | Domains to intercept (one per line) |
| `mocks/*/mappings/*.json` | WireMock stub response definitions |
| `app/src/*` | Application source code |
| `app/Dockerfile` | Application container build |
| `tests/playwright/*.spec.ts` | Playwright test specifications |

**Generated files (do NOT edit -- overwritten on start):**

| File | Generated by |
|------|-------------|
| `services/caddy.yml` | `devstack.sh` (from mock domains) |
| `services/wiremock.yml` | `devstack.sh` (from mock directories) |
| `caddy/Caddyfile` | `devstack.sh` (from mock domains + app type) |
| `domains.txt` | `devstack.sh` (aggregated from `mocks/*/domains`) |
| `certs/*` | cert-gen container (from `domains.txt`) |
| `tests/results/*` | Playwright test runner |

## File reading order

To understand this project, read in this order:

1. **`project.env`** -- what the project is called, what ports it uses, what database it has
2. **`docker-compose.yml`** -- what services are included (the include list is your stack)
3. **`services/app.yml`** -- how the application is built and configured
4. **`services/*.yml`** -- other infrastructure services
5. **`mocks/`** -- what external APIs are being mocked, with what responses
6. **`tests/playwright/`** -- what the tests expect
7. **`devstack.sh`** -- CLI commands (reference, not usually needed)

## Architecture in 30 seconds

```
Browser/Tests
     |
     v
  Caddy (:80/:443)    <-- reverse proxy, TLS termination
     |
     +--> App (:3000)  <-- your application
     |
     +--> WireMock     <-- mock API responses
          ^
          |
     App requests https://api.stripe.com/...
          |
     Docker DNS resolves api.stripe.com -> Caddy (network alias)
          |
     Caddy forwards to WireMock with X-Original-Host header
          |
     WireMock matches mocks/stripe/mappings/*.json
```

All services share the `devstack-internal` Docker network. Service names are hostnames: `app`, `db`, `redis`, `web` (Caddy), `wiremock`.

## How changes flow

### Changing application code
Edit files in `app/src/`. The app container mounts the source directory, so changes are reflected based on your app's hot-reload mechanism.

### Changing a mock response
Edit `mocks/<name>/mappings/*.json`, then run `./devstack.sh reload-mocks`. No restart needed.

### Adding a new mocked API
Run `./devstack.sh new-mock <name> <domain>`, then `./devstack.sh restart` (new domains need new certificates and DNS aliases).

### Adding an infrastructure service
Place a service YAML in `services/`, add an include line to `docker-compose.yml`, add any env vars to `project.env`, then `./devstack.sh restart`.

### Changing configuration
Edit `project.env`, then `./devstack.sh restart`.

## Commands reference

| Command | What it does |
|---------|-------------|
| `./devstack.sh start` | Build and start all services |
| `./devstack.sh stop` | Stop (preserve database data and caches) |
| `./devstack.sh stop --clean` | Stop and remove everything |
| `./devstack.sh restart` | Stop then start |
| `./devstack.sh restart --clean` | Full teardown then start |
| `./devstack.sh test [filter]` | Run Playwright tests |
| `./devstack.sh shell [service]` | Shell into a container (default: app) |
| `./devstack.sh status` | Container status and health |
| `./devstack.sh logs [service]` | Tail logs |
| `./devstack.sh mocks` | List configured mocks |
| `./devstack.sh new-mock <n> <d>` | Scaffold a new mock |
| `./devstack.sh reload-mocks` | Hot-reload mock mappings |
| `./devstack.sh record <mock>` | Record real API responses |
| `./devstack.sh apply-recording <mock>` | Apply recorded mappings |
| `./devstack.sh verify-mocks` | Verify mock DNS interception |

## Pitfalls

1. **`services/caddy.yml` and `services/wiremock.yml` are regenerated on every start.** Do not edit them. Your changes will be lost. If you need to change Caddy behavior, the generation logic is in `devstack.sh`.

2. **New mock domains require a full restart.** Adding a domain to `mocks/*/domains` needs new TLS certificates (SANs) and new Docker DNS aliases. `reload-mocks` only reloads WireMock mapping JSON -- it cannot add DNS entries or regenerate certs.

3. **Container-created files are root-owned.** Test results, recordings, and node_modules written by containers are owned by root on the host. Use `./devstack.sh stop --clean` to remove them, or use a Docker container: `docker run --rm -v $(pwd)/tests:/data alpine rm -rf /data/results/*`

4. **Playwright version must match the container image.** The version in `tests/playwright/package.json` must exactly match the Playwright Docker image version. Pin it without `^` or `~`.

5. **`stop` preserves volumes, `stop --clean` deletes them.** Database data, Redis caches, and MinIO objects persist across `stop`/`start` cycles. Use `--clean` only when you want a fresh start.

6. **Two mocked APIs with the same URL path need `X-Original-Host` disambiguation.** All mocked domains share one WireMock instance. If two APIs use the same method + path, add `"headers": { "X-Original-Host": { "equalTo": "api.specific.com" } }` to the mapping's request block. Caddy adds this header automatically.

7. **`project.env` is the single source of configuration.** All `${VAR}` references in service YAML files resolve from it via Docker Compose. The `.env` symlink points to `project.env`.
